# Тонкости при работе с Trino

## Отображение запросов в UI

Все запросы будут отображаться в UI даже если каталог скрыт от группы пользователей.

## Оптимизация запросов

Оптимизация запросов в Trino — искусство. На эту тему есть множество статей и видео. Но в целом, оптимизация запросов в
Trino не отличается от "_общих_" принципов оптимизации SQL-запросов:

- Предфильтрация данных: старайтесь фильтровать данные как можно раньше в запросе, чтобы уменьшить объем данных.
- Исключений конструкций `SELECT *`: выбирайте только те столбцы, которые вам действительно нужны.
- Корректное использование подзапросов и CTE: иногда подзапросы могут быть более эффективными, чем CTE, и наоборот. Это
  зависит от конкретного запроса и структуры данных.
- Своевременное использование временных таблиц: если запрос слишком сложный, можно разбить его на несколько этапов и
  использовать временные таблицы для хранения промежуточных результатов.
- Часть конструкций с `JOIN` можно заменить на подзапросы: они почти всегда выгоднее чем `JOIN`.
- Исключить "_ранние_" `JOIN`: аналитик сразу может объединить пользователей с их персональными данными (ФИО, Год
  рождения и пр.), а по факту это нужно только в конечной витрине и он с самого начала "_тащит_" этот большой кортеж
  между `JOIN`.

Важно понимать, как Trino выполняет запросы, какие операции являются дорогими и как можно минимизировать их
использование.

### Проталкивание запросов

Одна из небольших "_магий_" Trino — это проталкивание запросов.

Давайте рассмотрим на примере PostgreSQL.

В настоящее время Trino Connector SPI ограничивает типы операций, проталкиваемых в фильтры и проекции столбцов:

```sql
SELECT
  state,
  count(*)
FROM
  airport
WHERE
  country = 'US'
GROUP BY
  state;
```

В приведенном выше запросе Trino коннектор PostgreSQL создает SQL-запрос для проталкивания его в базу данных PostgreSQL:

```sql
SELECT
  state
FROM
  airport
WHERE
  country = 'US';
```

При проталкивании запросов коннектором СУРБД необходимо обратить особое внимание на два важных факта. Столбцы в списке
`SELECT` конкретно определяют то, что необходимо Trino. В рассматриваемом здесь примере необходим только столбец state
для обработки спецификатором `GROUP BY` в Trino. Также проталкивается фильтр `country = 'US'`, а это означает, что
дальнейшая обработка столбца country в Trino не требуется.

Обратите внимание: **агрегации не проталкиваются в PostgreSQL.**

Причина в том, что Trino не имеет возможности протолкнуть любую другую форму запроса, поэтому агрегации обязательно
должны выполняться в Trino. Это может оказаться преимуществом, поскольку Trino представляет собой распределенный
механизм обработки запросов, тогда как PostgreSQL таковым не является.

При необходимости протолкнуть дополнительную обработку в нижележащий источник СУРБД можно воспользоваться
представлениями (`views`). Если инкапсулировать операции обработки в представление PostgreSQL, то для Trino оно будет
выглядеть как таблица, а обработка будет выполнена в PostgreSQL.

Например, предположим, что вы создаете представление в PostgreSQL:

```sql
CREATE VIEW airline.airports_per_us_state AS
SELECT
  state, count(*) AS count_star
FROM
  airline.airport
WHERE
  country = 'US'
GROUP BY
  state;
```

Теперь можно просто запросить представление, и вся обработка проталкивается в PostgreSQL, поскольку созданное
представление выглядит как обычная таблица для Trino:

```sql
SELECT * FROM airports_per_us_state;
```

## Работа с ролевой моделью

Авторизация и аутентификация — один из важных элементов при работе с Trino.

Потому что у вас может быть условно заведён `root` в коннекторах для БД и вы можете получить интересные сайд-эффекты 😉.

Поэтому важно понимать, как работает ролевая модель в Trino и как она взаимодействует с ролевой моделью в источниках
данных, к которым вы подключаетесь.

Важно убедиться, что у вас есть правильные разрешения и роли, чтобы избежать неожиданных проблем с безопасностью и
доступом к данным.
